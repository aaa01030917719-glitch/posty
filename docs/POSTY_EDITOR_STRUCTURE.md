# POSTY_EDITOR_STRUCTURE.md

## 1. 콘텐츠 상세 / 글 작성 페이지의 핵심 원칙

Posty의 글 작성 화면은 콘텐츠 카드 1개를 중심으로 합니다.

예:
- 유튜브 쇼츠 0504
- 인스타 릴스 0504
- 스레드 콘텐츠 0504
- 블로그 5월 2주차

콘텐츠 상세 안에서 관리하는 작성 요소:
- 기본 원고
- 대본/씬
- 캡션
- 해시태그
- 썸네일 문구
- CTA
- 참고 링크
- 체크리스트
- 메모
- 작업 이력
- 일정 정보

중요:
원고, 대본, 캡션, 해시태그, 썸네일 문구는 각각 독립된 최상위 페이지가 아니라, 콘텐츠 상세 안의 작성 섹션으로 봅니다.

## 2. 기본 글 작성 화면 구조

기본 글 작성 화면은 매우 미니멀한 문서형 에디터를 기준으로 합니다.

디자인 방향:
- Notion/문서 편집기처럼 심플한 중앙 정렬 구조
- 불필요한 카드, 박스, border, 장식 최소화
- 글쓰기에 집중할 수 있는 넓은 여백
- 제목, 메타 정보, 본문 에디터 중심
- 의미 없는 구분선 사용 금지
- 배경은 white / very light gray 기반
- 텍스트는 또렷하게
- 버튼과 포인트 컬러는 최소 사용

기본 구성:
- 상단 breadcrumb 또는 콘텐츠 위치 정보
- 콘텐츠 제목
- 메타 정보
  - 채널
  - 상태
  - 작성일
  - 마지막 수정일
  - 업로드 예정일
- 기본 원고 작성 영역
- 본문 에디터
- 우측 상단 액션
  - 공유
  - 추가
  - 우측 패널 열기/닫기

표시 예:
작성일 26.05.04 · 마지막 수정 26.05.04 14:43

## 3. 우측 작성 패널 구조

우측 작성 패널은 기본 에디터 옆에 열리는 보조 작성 공간입니다.

역할:
기본 원고를 작성하다가, 대본, 캡션, 해시태그, 썸네일 문구 등 추가 작성물이 필요할 때 오른쪽 패널에서 선택해 작성합니다.

열림 방식:
- 오른쪽에서 슬라이드 인
- 닫을 수 있음
- 기본 글 작성 영역은 유지
- 패널이 열리면 본문 영역은 자연스럽게 줄어들거나, 화면 크기에 따라 overlay로 표시

우측 패널 상단 구성:
- 섹션 선택 드롭다운
  - 원고
  - 대본
  - 캡션
  - 해시태그
  - 썸네일 문구
  - 체크리스트
  - 메모
- 공유 버튼
- 추가 버튼
- 패널 닫기 버튼

사용 예:
사용자가 기본 원고를 작성하다가 대본도 필요하다고 판단하면, 우측 패널 상단 드롭다운에서 대본을 선택합니다. 그러면 같은 콘텐츠 안의 대본 작성 영역이 패널에 표시됩니다.

## 4. 드롭다운 섹션 전환 정책

우측 패널의 드롭다운은 콘텐츠 안의 작성 섹션을 전환하는 용도입니다.

드롭다운 항목 예:
- 원고
- 대본
- 캡션
- 해시태그
- 썸네일 문구
- 체크리스트
- 메모
- 작업 이력

동작 정책:
- 선택한 섹션만 패널에 표시
- 사용자가 추가하지 않은 섹션은 빈 화면으로 길게 노출하지 않음
- 필요한 섹션은 추가 버튼으로 활성화 가능
- 콘텐츠 형식에 따라 추천 섹션이 다름

예:

### 4.1 유튜브 쇼츠
- 기본 원고
- 대본/씬
- 캡션
- 해시태그
- 썸네일 문구

### 4.2 스레드
- 기본 원고
- 후킹 문장
- 키워드
- 캡션 또는 본문

### 4.3 블로그
- 기본 원고
- 목차
- SEO 키워드
- CTA
- 참고 링크

### 4.4 카드뉴스
- 슬라이드별 문구
- 캡션
- 디자인 체크리스트

## 5. 대본/씬 작성 정책

대본은 콘텐츠 안의 보조 작성 섹션입니다.

대본 패널에는 여러 개의 씬을 둘 수 있습니다.

씬 항목:
- 씬 번호
- 씬 제목
- 씬 내용
- 삭제 버튼
- 순서 변경 가능성
- 자동 저장 상태

대본 패널 상단의 이름은 수정 가능해야 합니다.

예:
- 기본값: 대본
- 사용자가 클릭해서 수정:
  - 숏폼 대본
  - 릴스 대본
  - 촬영 순서
  - 장면 구성

## 6. /scripts 라우팅 역할

/scripts는 원고 작성의 본진이 아닙니다.

역할:
- 원고 작업이 있는 콘텐츠를 모아보는 보조 목록 페이지
- 작성 중 원고
- 최종본 대기
- 캡션 미작성
- 대본 있음
- 수정 필요

/scripts에서 항목을 클릭하면 실제 편집은 콘텐츠 상세로 이동합니다.

예:
- /content/[id]?section=script
- /content/[id]?section=scenes

즉, 실제 원고와 대본의 소유자는 콘텐츠 카드입니다.

## 7. 공유 기능 위치 정책

추후 대표/클라이언트 컨펌을 위해 읽기 전용 공유 URL 기능이 필요합니다.

글 작성 페이지에서는 우측 상단에 공유 버튼 위치를 고려합니다.

정책:
- 공유받은 사람은 로그인 없이 읽기 가능
- 수정 불가
- 읽기 전용
- 공유 대상:
  - 콘텐츠 카드
  - 원고
  - 대본
  - 캘린더 일정
- 지금 당장 구현하지 않음
- UI 위치만 고려

## 8. 디자인 방향

기본 글 작성 화면:
- 아주 미니멀
- 문서형
- 넓은 여백
- 흰 배경
- 불필요한 디테일 제거
- 텍스트 중심
- 의미 없는 선/카드/border 금지

우측 작성 패널:
- 상단에 드롭다운과 액션 버튼 배치
- 패널 내부는 글 작성 보조 공간
- 너무 화려하지 않게
- 기능은 명확하게
- 패널 닫기 가능

색상:
- Airbnb md color palette 유지
- primary #ff385c는 CTA, active, 선택 상태에만 제한적으로 사용
- 배경은 #ffffff / #f7f7f7 / #f2f2f2 중심
- 텍스트는 #222222 / #3f3f3f 중심
- border는 필요한 경우에만 #dddddd 또는 #ebebeb 사용

## 9. 리치 텍스트 에디터 후속 구현 정책

Posty의 글 작성 페이지는 장기적으로 리치 텍스트 에디터 기능을 지원합니다.

예상 기능:
- 글자 크기 변경
- 굵게
- 이탤릭
- 밑줄
- 리스트
- 번호 리스트
- 링크 삽입
- 인용
- 구분선
- 이미지/파일 첨부
- 되돌리기/다시 실행

현재 단계 정책:
- 지금은 실제 리치 텍스트 기능을 구현하지 않습니다.
- 글 작성 화면에는 에디터 툴바가 들어갈 수 있는 위치만 고려합니다.
- 기본 원고 작성은 단순 텍스트 입력 또는 placeholder editor 영역으로 충분합니다.
- Tiptap, Lexical, ProseMirror 같은 에디터 라이브러리 도입은 별도 작업으로 판단합니다.
- 저장 포맷은 후속 구현 시 plain text / markdown / json rich text 중 결정합니다.

금지:
- 이번 단계에서 에디터 라이브러리 설치 금지
- 굵게/이탤릭/글자 크기 변경 기능 구현 금지
- 저장 포맷 변경 금지
- DB schema 변경 금지

## 10. 우측 작성 패널 너비 조절 후속 구현 정책

Posty의 우측 작성 패널은 장기적으로 사용자가 너비를 조절할 수 있도록 확장할 수 있습니다.

목적:
- 기본 원고와 보조 작성 섹션을 동시에 볼 때 사용자가 작업 방식에 맞게 공간을 조절할 수 있게 하기 위함
- 대본/씬처럼 긴 내용을 작성할 때 패널을 넓게 사용할 수 있게 하기 위함

예상 동작:
- 패널 왼쪽 edge에 resize handle 제공
- 사용자가 드래그해서 패널 너비 조절
- 최소 너비와 최대 너비 제한
- 데스크톱에서만 제공
- 모바일에서는 전체 화면 overlay로 표시하고 resize는 제공하지 않음
- 사용자가 설정한 너비는 추후 localStorage 또는 사용자 설정으로 저장 가능

초기 구현 정책:
- 현재 단계에서는 실제 resize 기능을 구현하지 않습니다.
- 첫 구현은 고정 너비 패널로 진행합니다.
- 권장 초기 패널 너비는 360px ~ 420px 사이입니다.
- resize handle 위치만 향후 확장 가능성을 고려합니다.
- 실제 drag resize, 너비 저장, 반응형 세부 처리는 후속 작업으로 분리합니다.

금지:
- 이번 단계에서 resize 기능 구현 금지
- drag 이벤트 로직 구현 금지
- localStorage 저장 구현 금지
- 패널 레이아웃 실제 코드 수정 금지

## 11. 이번 문서에서 구현하지 않는 것

이번 작업에서는 아래를 구현하지 않습니다.

- /content/[id] route 생성
- 실제 Editor 컴포넌트 구현
- 우측 패널 컴포넌트 구현
- 드롭다운 구현
- 대본/씬 DB 구현
- 공유 기능 구현
- /scripts 구조 변경
- UI 리디자인
- DB schema 수정
