# CODEX_RULES.md

포스티(Posty) 프로젝트에서 Codex가 작업할 때 반드시 지켜야 하는 운영 규칙입니다.

목표는 다음 3가지입니다.

1. 토큰 절약
2. 같은 파일을 반복해서 읽는 문제 방지
3. 중복 코드, 더러운 코드, 예기치 못한 오류 방지

이 파일은 프로젝트 루트에 두고, Codex 작업을 시작할 때 항상 먼저 읽게 합니다.

---

## 0. 가장 중요한 원칙

포스티는 새 프로젝트이므로, 이전 프로젝트에서 생겼던 문제를 반복하지 않는다.

이전 문제:
- 작업 전후로 불필요하게 많은 파일을 계속 읽음
- 같은 파일을 여러 번 반복 확인함
- 기존 CSS와 컴포넌트를 재사용하지 않고 비슷한 코드를 새로 만듦
- 전역 CSS가 계속 커짐
- 작은 수정인데도 넓은 범위를 건드림
- dirty 파일과 작업 파일이 섞임
- 사용자가 원하지 않은 commit, push, 자동 수정이 발생함
- 화면 확인 없이 “수정 완료”라고 보고함

포스티에서는 아래 원칙을 기본값으로 한다.

- 필요한 파일만 읽는다.
- 이미 읽은 파일은 같은 세션에서 다시 읽지 않는다.
- 수정 범위는 요청받은 부분으로 제한한다.
- 기존 디자인 시스템과 컴포넌트를 먼저 찾고 재사용한다.
- 새 CSS, 새 컴포넌트, 새 유틸은 정말 필요할 때만 만든다.
- 수정 전 `git status`를 확인한다.
- 수정 후 변경 파일, 검증 결과, 남은 리스크만 간결하게 보고한다.

## 1. 프로젝트 경로 규칙

작업 시작 시 반드시 사용자가 지정한 포스티 프로젝트 경로에서만 작업한다.

- 사용자가 경로를 명확히 말하지 않았다면 파일 수정 전 경로를 확인한다.
- 임의로 다른 프로젝트 경로를 사용하지 않는다.
- 특히 이전 프로젝트 경로인 `C:\daylog_codex`, `C:\linkko`, `C:\codex bridge`를 포스티 작업에 사용하지 않는다.
- 경로가 다르면 즉시 멈추고 사용자에게 확인한다.

작업 시작 예시:

```powershell
cd /d C:\posty
git status --short
```

단, 실제 경로가 다르면 사용자가 말한 경로를 따른다.

## 2. 작업 시작 체크리스트

파일을 수정하기 전에 반드시 아래 순서로 확인한다.

1. `git status --short`
   - 이미 변경된 파일이 있는지 확인한다.
   - 사용자가 만든 변경분과 Codex 작업분을 섞지 않는다.

2. 요청 범위 확인
   - 사용자가 요청한 화면, 기능, 파일만 대상으로 삼는다.
   - 요청에 없는 리디자인, 리팩터링, 정리 작업은 하지 않는다.

3. 관련 파일 최소 검색
   - 전체 파일을 무작정 읽지 않는다.
   - 먼저 `rg`로 후보 파일을 좁힌다.
   - 후보 파일 중 실제 수정 가능성이 높은 파일만 읽는다.

4. 기존 구현 확인
   - 비슷한 UI, 컴포넌트, 토큰, 유틸이 있는지 먼저 찾는다.
   - 기존 구현이 있으면 새로 만들지 않고 재사용한다.

5. 짧은 작업 계획 작성
   - 수정 파일 예상 목록
   - 수정 방향
   - 확인 방법
   - 작업하지 않을 범위

## 3. 토큰 절약 규칙

Codex는 작업 중 다음을 금지한다.

- 저장소 전체를 무작정 읽기
- 같은 파일을 같은 목적으로 반복해서 읽기
- 긴 파일 전체를 매번 출력하기
- 빌드 로그 전체를 보고에 붙여넣기
- 무관한 파일 diff를 길게 설명하기
- “혹시 몰라서” 넓은 범위 검색 반복하기
- 변경 없는 파일을 다시 읽기

권장 방식:

```powershell
rg "검색어" .
rg "컴포넌트명|클래스명|route명" app components lib
Get-Content path\to\file.tsx -TotalCount 220
git diff -- path\to\file.tsx
```

긴 파일은 필요한 범위만 확인한다.

```powershell
Get-Content path\to\file.tsx | Select-Object -Skip 120 -First 120
```

같은 파일을 다시 읽어야 하는 경우:
- 파일이 수정되었을 때
- 다른 검색 결과로 인해 새로운 구간 확인이 필요할 때
- 빌드 오류가 해당 파일의 다른 줄을 가리킬 때

그 외에는 다시 읽지 않는다.

---

## 4. 파일 수정 범위 규칙

기본 원칙:

- 요청받은 기능과 직접 관련된 파일만 수정한다.
- 단순 UI 수정은 UI 파일과 필요한 스타일 파일만 수정한다.
- 한 번에 큰 리팩터링을 하지 않는다.
- “겸사겸사 정리”를 하지 않는다.
- 새 기능과 리디자인을 섞지 않는다.
- 조사 요청이면 절대 파일을 수정하지 않는다.

금지:
- 요청 없이 폴더 구조 변경
- 요청 없이 파일명 변경
- 요청 없이 대량 삭제
- 요청 없이 패키지 설치
- 요청 없이 dependency 업데이트
- 요청 없이 전역 스타일 대규모 변경
- 요청 없이 `lint --fix`, `prettier --write .`, format all 실행
- 요청 없이 commit/push

## 5. 디자인 시스템 우선 규칙

포스티는 처음부터 중복 CSS와 더러운 코드를 방지해야 한다.

UI 작업 시 우선순위:

1. 기존 디자인 토큰 사용
2. 기존 공통 컴포넌트 사용
3. 기존 패턴 확장
4. 필요한 경우에만 새 컴포넌트 생성
5. 정말 필요한 경우에만 새 전역 CSS 추가

새 스타일을 만들기 전에 반드시 확인할 것:

- 이미 같은 역할의 버튼이 있는가?
- 이미 같은 역할의 카드가 있는가?
- 이미 같은 역할의 필터칩, 탭, 배지, 드롭다운이 있는가?
- 이미 같은 여백, radius, shadow, border 토큰이 있는가?
- 이미 같은 레이아웃 패턴이 있는가?

금지:
- 페이지마다 비슷한 버튼 스타일을 새로 만들기
- `globals.css`에 임시 클래스 계속 추가하기
- inline style 남발하기
- 색상 hex 값을 무분별하게 직접 넣기
- 같은 UI를 다른 이름으로 중복 구현하기
- 디자인 시스템을 우회해서 빠르게만 구현하기

허용:
- 디자인 시스템에 없는 패턴이 명확할 때만 새 컴포넌트 제안
- 새 컴포넌트를 만들 경우 사용 위치와 이유를 보고
- 전역 CSS가 필요하면 왜 전역이어야 하는지 보고

## 6. Figma / UI Kit 관련 규칙

포스티는 UI Kit 기반으로 UI 시스템을 만들 수 있다.

단, Codex는 Figma 화면을 임의로 추측하지 않는다.

- Figma에서 확인 가능한 에셋, 토큰, 컴포넌트가 제공된 경우에만 반영한다.
- 사용자가 캡처, export, 코드, 토큰, 설명을 제공하지 않았다면 디자인을 상상해서 구현하지 않는다.
- UI Kit를 적용할 때는 먼저 “토큰 → 컴포넌트 → 화면” 순서로 정리한다.
- 화면부터 급하게 만들고 나중에 디자인 시스템을 맞추는 방식은 피한다.
- 아이콘, 컬러, spacing, radius, typography는 한 곳에서 관리되게 한다.

권장 구조 예시:

```text
styles/
  tokens.css

components/
  ui/
    Button.tsx
    Card.tsx
    Badge.tsx
    FilterChip.tsx
    Modal.tsx
    Input.tsx
```

실제 구조는 프로젝트 상태에 맞춘다.

## 7. 컴포넌트 생성 규칙

새 컴포넌트는 아래 조건을 만족할 때만 만든다.

- 같은 UI가 2곳 이상에서 반복될 가능성이 높다.
- 페이지 내부에 두면 코드가 과도하게 길어진다.
- 디자인 시스템의 기본 요소로 재사용할 가치가 있다.
- 기존 컴포넌트로 해결하기 어렵다.

새 컴포넌트 생성 시 지켜야 할 것:

- 이름은 역할 중심으로 짓는다.
- 페이지 전용이면 `components/[page]` 아래에 둔다.
- 공통이면 `components/ui` 또는 프로젝트의 공통 컴포넌트 위치에 둔다.
- props는 최소화한다.
- 스타일 변형은 variant로 관리한다.
- 불필요한 상태와 effect를 넣지 않는다.

금지:
- 한 번 쓰는 UI를 무조건 공통 컴포넌트로 만들기
- 공통 컴포넌트에 페이지 전용 로직 넣기
- props를 과도하게 늘리기
- 기존 컴포넌트를 복사해서 이름만 바꾸기

## 8. CSS / 스타일 규칙

CSS는 최대한 작게 유지한다.

우선순위:

1. 기존 토큰
2. 기존 utility class
3. 기존 컴포넌트 variant
4. module 또는 컴포넌트 근처 스타일
5. 마지막 수단으로 전역 CSS

전역 CSS 추가 전 확인:

- 이 스타일이 앱 전체에서 재사용되는가?
- 기존 토큰으로 표현할 수 없는가?
- 특정 페이지 전용 스타일이 아닌가?
- 이름이 너무 일반적이지 않은가?

금지:
- `.box`, `.card`, `.button`, `.title` 같은 충돌 위험 이름
- 기존 클래스와 거의 같은 새 클래스
- 임시 이름 그대로 남기기
- 사용하지 않는 CSS 남기기
- `!important` 남발
- z-index 임의 증가

## 9. 상태 관리 규칙

상태는 가장 가까운 곳에 둔다.

- 한 화면에서만 쓰는 상태는 해당 컴포넌트에 둔다.
- 여러 화면에서 공유되는 상태만 store/context로 올린다.
- localStorage/sessionStorage는 사용 목적이 명확할 때만 쓴다.
- 사용자가 선택한 UI 설정을 유지해야 하는 경우 key 이름을 명확히 한다.
- 서버 상태와 클라이언트 상태를 섞지 않는다.

localStorage 사용 시:
- key 이름은 프로젝트 prefix를 붙인다.
- 예: `posty.calendar.visibleGroups`
- 초기값, 없는 값, 깨진 값에 대한 fallback을 둔다.

## 10. 라우팅 규칙

라우팅은 임의로 바꾸지 않는다.

- 기존 route 구조를 먼저 확인한다.
- 상세 페이지가 어떤 섹션 소속인지 먼저 정한다.
- 사이드바 active 정책과 뒤로가기 버튼 정책을 함께 확인한다.
- 비슷한 상세 route를 재사용할 때 사이드바 active 충돌을 조심한다.

금지:
- 임시로 sidebar 조건만 추가해서 active 문제를 덮기
- 상세 페이지 흐름이 불명확한 상태에서 버튼 경로만 바꾸기
- 다른 섹션 route를 빌려 쓰면서 네비게이션 정책을 정하지 않기

## 11. 데이터 / API 규칙

데이터 구조 변경은 신중하게 한다.

- mock data, local data, API schema를 먼저 구분한다.
- 실제 API가 있는 경우 응답 타입을 확인한다.
- 타입 변경은 사용처 전체를 검색한 후 진행한다.
- nullable 값, 빈 배열, undefined 상태를 처리한다.
- 임시 데이터는 명확히 표시하고 나중에 제거할 수 있게 한다.

금지:
- 타입 오류를 `any`로 덮기
- optional 값을 강제로 단언하기
- API 응답을 추측해서 필드 추가하기
- mock 데이터 구조와 실제 데이터 구조를 섞기

## 12. 오류 방지 규칙

수정 중 오류가 나면 다음 순서로 대응한다.

1. 오류 메시지의 파일과 줄 번호 확인
2. 해당 파일의 해당 구간만 확인
3. 원인 파악
4. 최소 수정
5. 같은 유형의 사용처 검색
6. 다시 검증

금지:
- 원인 파악 없이 대규모 수정
- 오류를 숨기기 위한 타입 단언
- 빌드가 깨졌는데 완료 보고
- 실패 로그를 무시하고 push
- 사용자가 요청하지 않은 우회 구현

## 13. 검증 규칙

작업 성격에 따라 최소 검증을 한다.

문서/조사만 한 경우:
- 파일 수정 없음 확인
- `git status --short` 확인

UI 수정:
- 관련 화면 실행 확인
- 반응형 영향이 있으면 모바일 폭도 확인
- 기존 주요 화면이 깨지지 않는지 확인
- 가능하면 스크린샷 또는 확인 경로 보고

로직 수정:
- 관련 기능 수동 확인
- 타입 체크 또는 빌드 확인
- edge case 확인

빌드:
```powershell
npm run build
```

단, 프로젝트에 별도 검증 명령이 있으면 그 명령을 따른다.

금지:
- 검증하지 않고 “완료”라고 말하기
- 빌드 실패 상태에서 commit/push
- 자동 수정 명령으로 문제를 덮기

## 14. Git 규칙

작업 전:
```powershell
git status --short
```

작업 후:
```powershell
git diff --stat
git diff -- path\to\changed-file
git status --short
```

스테이징:
- 요청 작업과 직접 관련된 파일만 stage 한다.
- `git add .` 금지
- README, 설정, lockfile 등 무관 파일은 제외한다.
- 사용자가 만든 dirty 파일은 건드리지 않는다.

커밋:
- 사용자가 명시적으로 요청한 경우에만 한다.
- 커밋 전 빌드 또는 필요한 검증을 통과해야 한다.
- 커밋 메시지는 작업 내용을 좁게 표현한다.

예시:
```text
feat: add posty design tokens
fix: preserve calendar filter visibility
style: refine link list item layout
docs: add codex rules
```

푸시:
- 사용자가 명시적으로 요청한 경우에만 한다.
- push 전 현재 브랜치 확인
- push 전 `git status --short` 확인

## 15. 보고 형식

작업 보고는 짧고 정확하게 한다.

수정 완료 보고 형식:

```text
완료했습니다.

수정 파일:
- 파일1
- 파일2

변경 내용:
- 핵심 변경 1
- 핵심 변경 2

확인:
- npm run build 성공
- 화면 확인 완료

주의/남은 작업:
- 필요한 경우에만 작성
```

조사 보고 형식:

```text
조사 결과입니다.

확인한 범위:
- 경로/파일/검색어

발견한 위치:
- 파일: 내용 요약
- 파일: 내용 요약

판단:
- 중복 여부
- 위험 요소
- 추천 순서

파일 수정:
- 없음
```

오류 보고 형식:

```text
진행 중 문제가 있습니다.

문제:
- 무엇이 실패했는지

원인 추정:
- 확인한 근거

수정하지 않은 이유:
- 사용자 확인이 필요한 경우

추천:
- 다음에 할 일
```

## 16. 사용자 지시 우선순위

사용자의 현재 지시가 가장 우선이다.

우선순위:

1. 현재 대화의 직접 지시
2. 프로젝트 규칙 파일
3. 기존 프로젝트 구조
4. 일반 개발 관행

사용자가 “조사만”이라고 하면:
- 파일 수정 금지
- commit 금지
- push 금지
- 자동 수정 금지

사용자가 “토큰절약”이라고 하면:
- 긴 설명 금지
- 필요한 명령만 실행
- 변경 파일과 결과만 보고

사용자가 “정확하게”라고 하면:
- 추측 금지
- 확인한 내용과 추정 내용을 분리

사용자가 “간결하게”라고 하면:
- 불필요한 배경 설명 금지
- 바로 붙여넣을 수 있는 지시문 중심

## 17. 작업 분리 규칙

서로 다른 성격의 작업은 섞지 않는다.

분리해야 하는 작업 예시:
- UI 리디자인
- 디자인 시스템 정리
- 버그 수정
- 라우팅 정책 변경
- 상태 저장 기능
- 데이터 구조 변경
- 빌드 오류 해결
- 문서 작성
- commit/push

한 작업 중 발견한 다른 문제는 즉시 수정하지 말고 “후속 작업”으로 보고한다.

예시:

```text
후속으로 분리할 작업:
- 사이드바 active 정책 정리
- 공통 FilterChip 컴포넌트화
- 중복 카드 스타일 정리
```

## 18. 임시 파일 / 로그 규칙

임시 파일은 저장소 안에 만들지 않는다.

허용:
- `%TEMP%` 아래 임시 로그
- 작업 후 삭제되는 임시 파일

금지:
- 프로젝트 루트에 임시 로그 남기기
- `.codex-*` 파일 남기기
- 인증 토큰 파일 남기기
- 테스트용 이미지/JSON을 루트에 방치하기

작업 종료 전:
```powershell
git status --short
```

예상하지 못한 파일이 생겼다면 삭제하거나 사용자에게 보고한다.

## 19. 성능 / 품질 규칙

기능 구현 시 기본적으로 확인할 것:

- 불필요한 re-render를 만들지 않았는가?
- 무거운 계산을 렌더링마다 하지 않는가?
- 리스트 key가 안정적인가?
- 모바일에서 레이아웃이 깨지지 않는가?
- 접근성 기본값이 있는가?
- 버튼/링크 역할이 올바른가?
- 로딩, 빈 상태, 오류 상태가 있는가?

단, 작은 UI 수정에서 과도한 최적화는 하지 않는다.

---

## 20. 접근성 기본 규칙

새 UI를 만들 때 최소한 아래를 지킨다.

- 버튼은 실제 `button` 사용
- 링크는 실제 `a` 또는 라우터 Link 사용
- 아이콘만 있는 버튼은 accessible label 제공
- input은 label 또는 aria-label 제공
- focus 상태를 제거하지 않음
- 색상만으로 상태를 구분하지 않음

---

## 21. 모바일 규칙

포스티는 모바일에서도 자연스럽게 보여야 한다.

UI 수정 시 확인:
- 작은 화면에서 줄바꿈이 자연스러운가?
- 버튼이 너무 작지 않은가?
- 고정 영역이 콘텐츠를 가리지 않는가?
- 가로 스크롤이 생기지 않는가?
- bottom nav, header, sidebar와 충돌하지 않는가?

## 27. 완료 기준

Codex는 아래 기준을 만족해야 완료 보고를 할 수 있다.

- 요청 범위 안에서만 수정했다.
- 관련 파일만 변경했다.
- 기존 dirty 파일을 건드리지 않았다.
- 중복 컴포넌트/CSS를 만들지 않았다.
- 필요한 검증을 했다.
- 실패한 검증이 있으면 숨기지 않고 보고했다.
- commit/push는 사용자가 요청한 경우에만 했다.
- 남은 리스크와 후속 작업을 분리해서 보고했다.
